# 外卖业务底座（第一期）设计稿

**日期：** 2026-04-20  
**阶段：** 第一阶段 / 业务底座  
**定位：** 将当前 smart_order 升级为一个可作为 agent 应用项目底座的类美团外卖系统，优先完成用户侧交易主链路，为后续智能助手与高质量 RAG 提供结构化业务上下文。

---

## 1. 目标

第一期目标不是一次性做完整外卖平台，而是先完成一个可演示、可扩展、可继续承接 agent 能力的业务底座。该底座需要具备接近真实外卖产品的核心用户体验：

- 用户注册、登录、鉴权刷新
- 浏览商家与菜品
- 跨商家加入购物车
- 管理收货地址
- 提交结算并生成订单
- 完成模拟支付
- 查看订单列表与订单状态
- 对配送范围、配送费用、配送时长进行规则或地图能力校验

第一期同时要为第二期智能助手建设预留清晰接入点，使后续助手能够基于商家、菜品、用户、地址、购物车、订单等上下文进行问答、推荐、辅助点餐和订单解释。

---

## 2. 第一期开闭范围

### 2.1 第一期开做内容

第一期只做 **用户侧外卖平台主链路**，并以“准生产演示”而不是“最小 demo”为目标。系统需覆盖以下业务域：

1. **认证与用户能力**
   - 账号密码注册
   - 登录
   - JWT Access Token
   - Refresh Token 刷新
   - 获取当前登录用户

2. **商家与菜品目录能力**
   - 商家列表
   - 商家详情
   - 菜品列表与分类展示
   - 基础搜索与筛选能力

3. **购物车能力**
   - 跨商家加购
   - 修改数量
   - 删除菜品
   - 按商家分组查看购物车内容

4. **地址与配送能力**
   - 地址新增、编辑、删除、默认地址设置
   - 配送范围校验
   - 配送费预估
   - 配送时长预估

5. **交易能力**
   - 结算预览
   - 一次提交，多商家拆单
   - 生成父单与子单
   - 模拟支付
   - 订单列表
   - 订单详情
   - 订单状态流转

6. **种子数据能力**
   - 将商家、菜品、地理位置等数据写入数据库
   - 构造较真实的数据量与数据分布
   - 替代当前 `.env` 中写死的单商家地址方式

7. **Agent 接入预留能力**
   - 为后续智能助手统一暴露业务上下文读取接口
   - 不在本期实现真正的聊天与 RAG 流程

### 2.2 第一期开不做内容

为了控制范围，以下能力明确不纳入第一期：

- 商家运营后台
- 骑手调度或运力分配
- 真实支付接入
- 优惠券、满减、促销、退款、售后
- 智能助手 UI 对话能力
- RAG 构建与向量召回实现
- 商家自主入驻与审核流程

这样可以把第一期聚焦在“真实交易主链路 + 为 agent 铺路”的基础设施上，而不是把时间消耗在过早的扩展功能上。

---

## 3. 总体架构

第一期建议延续现有项目的前后端分层，但把业务能力从原始 demo 升级为标准电商/外卖式模块：

- **前端层**：用户侧页面，负责商家浏览、加购、结算、地址管理、订单展示、登录注册
- **API 层**：按业务域拆分接口（Auth、Catalog、Address、Cart、Order、Agent Context）
- **应用服务层**：封装结算、拆单、配送校验、支付模拟、订单流转等业务逻辑
- **数据访问层**：负责用户、地址、商家、菜品、购物车、订单等实体读写
- **数据库层**：存储结构化业务数据
- **种子数据脚本层**：生成真实感更强的基础商家与菜品样本

本期的核心原则是：

1. **交易链路先完整**，再接智能能力
2. **结构化数据先建对**，后续 RAG 才能做强
3. **多商家结算模型一次设计对**，避免后期推翻
4. **Agent 能力通过独立上下文接口接入**，不和核心交易 API 混在一起

---

## 4. 核心模块划分

### 4.1 认证与用户模块

职责：

- 处理注册、登录、刷新 token、当前用户信息
- 管理用户身份与权限边界
- 为购物车、地址、订单等能力提供登录上下文

设计决策：

- 第一阶段采用 **账号密码 + JWT / Refresh Token**
- 这样既能体现完整认证链路，也方便前端 SPA 场景
- 后续助手如果需要读取用户上下文，也可直接复用鉴权体系

### 4.2 商家与菜品目录模块

职责：

- 提供商家列表与商家详情
- 管理商家与菜品的层级关系
- 提供分类浏览、推荐位、基础搜索/筛选的接口基础

设计决策：

- 不做商家后台，而是由种子数据和脚本维护目录数据
- 第一阶段重点是“展示像真实平台”，不是“让平台运营人员手工录入”

### 4.3 购物车模块

职责：

- 支持跨商家购物车
- 维护用户当前购物意图
- 在结算前按商家分组组织商品

设计决策：

- 允许多个商家的商品同时存在于购物车中
- 但进入结算后，系统内部必须按商家拆单处理
- 购物车返回结构要天然支持按商家聚合，方便前端展示和后续助手分析

### 4.4 地址与配送模块

职责：

- 管理用户收货地址
- 判断商家是否可送达指定地址
- 估算配送时间和配送费用

设计决策：

- 第一期做“真实地址管理 + 配送范围校验 + 费用/时长预估”
- 不做骑手调度
- 如果可复用地图能力则保留接口，不可复用时可先由规则引擎或距离算法实现

### 4.5 交易模块

职责：

- 处理结算、拆单、创建订单、支付记录、订单状态流转
- 对多商家混合结算提供统一抽象

设计决策：

- 采用 **父单 + 子单** 模型
- 用户视角是“一次结算”
- 系统履约视角是“按商家拆成多个子订单”
- 为后续解释型助手和订单推荐打基础

### 4.6 种子数据模块

职责：

- 向数据库插入多个商家、多类菜品、多组地理位置与展示数据
- 构造更像真实平台的数据分布

设计决策：

- 目标数据量建议为 **20~50 个商家**、每商家 **10~30 个菜品**
- 覆盖多个商圈、不同菜系、不同价格带、不同配送半径
- 让前端、搜索、后续 RAG 都有足够数据基础

### 4.7 Agent 接入预留模块

职责：

- 统一输出智能助手后续需要的结构化业务上下文
- 减少将来助手直接拼装多张业务表的复杂度

设计决策：

- 第一期不做聊天能力，但必须把接口边界先设计好
- 后续助手可基于这些结构化接口叠加向量召回与推理能力

## 5. 核心数据模型

第一期建议的数据模型如下。

### 5.1 用户与地址

- `User`
  - 用户基础信息
  - 登录凭证摘要
  - 账户状态

- `UserAddress`
  - 归属用户
  - 联系人、手机号
  - 地址标签（家/公司/学校等）
  - 详细地址与门牌
  - 经纬度
  - 是否默认地址

### 5.2 商家与菜品

- `Merchant`
  - 商家名称
  - 商家地址
  - 经纬度
  - 营业状态
  - 配送配置
  - 评分、展示图、描述等展示字段

- `DishCategory`
  - 商家下的分类
  - 分类名与排序

- `Dish`
  - 所属商家与分类
  - 菜品名、价格、图片、描述
  - 标签（招牌、辣、适合双人等）
  - 上下架状态
  - 推荐状态

### 5.3 购物车

- `Cart`
  - 归属用户
  - 当前购物车聚合容器

- `CartItem`
  - 所属用户 / 所属购物车
  - 商家 ID
  - 菜品 ID
  - 数量
  - 当前加购时的价格快照

### 5.4 订单与支付

- `CheckoutOrder`（父单）
  - 一次结算对应一个父单
  - 保存用户、地址快照、商品总价、配送总价、应付总价
  - 保存支付状态与整体订单状态

- `MerchantOrder`（子单）
  - 一个父单下按商家拆出多个子单
  - 保存商家信息、子单金额、配送费、履约状态

- `OrderItem`
  - 子单下的菜品快照
  - 保留菜品名、价格、数量等，避免后续菜品修改污染历史订单

- `PaymentRecord`
  - 模拟支付记录
  - 渠道类型（mock）
  - 支付结果
  - 请求号与时间

- `DeliveryQuote`
  - 结算时生成的配送校验快照
  - 记录是否可送达、配送费、预估时长
  - 供订单创建时固化历史状态

## 6. 关键业务规则

### 6.1 多商家混合结算

用户可以把多个商家的商品同时加入购物车。进入结算时，系统先按商家分组，然后分别对每个商家执行：

1. 菜品有效性校验
2. 库存与状态校验
3. 地址配送范围校验
4. 配送费与时长估算

所有商家都通过后，系统再统一生成：

- 一个父单 `CheckoutOrder`
- 多个子单 `MerchantOrder`
- 各子单下的 `OrderItem`
- 对应的 `DeliveryQuote` 快照

### 6.2 订单创建一致性

第一期采用 **全有或全无** 策略：

- 如果任一商家无法配送或校验失败，则整次提交失败
- 不允许一次提交中部分商家成功、部分商家失败

这样能大幅降低首期复杂度，避免前端、支付、订单状态出现不一致。

### 6.3 支付策略

第一期采用 **模拟支付**：

- 提交结算后生成待支付父单/子单
- 执行模拟支付接口
- 支付成功则父单更新为已支付，子单进入履约初始状态
- 支付失败则保持待支付或失败状态

### 6.4 订单状态流转

父单建议状态：

- `pending_payment`
- `paid`
- `completed`
- `cancelled`

子单建议状态：

- `pending_payment`
- `paid`
- `preparing`
- `delivering`
- `completed`
- `cancelled`

第一期不做真实骑手调度，因此 `delivering` 由系统模拟推进或管理脚本推进。

## 7. API 边界

### 7.1 Auth API

负责：

- 注册
- 登录
- 刷新 token
- 获取当前用户

### 7.2 Catalog API

负责：

- 商家列表
- 商家详情
- 菜品列表
- 菜品详情
- 基础搜索与筛选

### 7.3 Address API

负责：

- 地址新增、编辑、删除
- 地址列表
- 默认地址切换

### 7.4 Cart API

负责：

- 查询购物车
- 加购
- 修改数量
- 删除菜品
- 按商家聚合返回购物车视图

### 7.5 Order API

负责：

- 结算预览
- 提交订单
- 模拟支付
- 查询订单列表
- 查询订单详情

### 7.6 Agent Context API

负责：

- 向后续智能助手暴露结构化业务上下文
- 提供用户地址、购物车、最近订单、商家/菜品等统一读取入口

本期该 API 只做预留，不直接提供聊天交互。

第一期前端建议至少包含以下页面：

1. **登录 / 注册页**
   - 支持账号密码注册与登录

2. **首页 / 商家列表页**
   - 展示商家卡片
   - 支持基础筛选

   - 展示分类与菜品列表
   - 支持加入购物车

4. **购物车 / 结算页**

   - 地址增删改查
   - 默认地址设置

6. **订单列表 / 订单详情页**

   - 展示按商家拆分后的子单
   - 展示支付状态与履约状态

7. **助手入口占位区**
   - 页面右下角保留未来助手挂载区域

## 9. 种子数据策略

当前项目中的商家地址不应继续依赖 `.env` 的单点写死模式。第一期应将真实感数据迁移到数据库和种子脚本中，至少满足：

- 20~50 个商家
- 每商家 10~30 个菜品
- 覆盖中餐、轻食、咖啡甜点、炸鸡汉堡、面食等多个品类
- 商家分布在多个商圈
- 不同商家拥有不同配送半径、配送费、起送门槛
- 菜品拥有不同标签、价格带与推荐位

这套数据将成为后续：

- 平台首页展示
- 商家搜索与筛选
- 配送范围判断
- 智能推荐与问答
- RAG 向量化与召回

的统一基础。

## 10. 为什么这套底座适合后续 Agent / RAG

后续要做一个能写进简历的智能外卖助手，关键不只是“能回答问题”，而是要让助手建立在真实业务上下文之上。第一期底座完成后，第二期助手就可以处理以下类型的问题：

- 当前地址有哪些商家可送达
- 两个人、50 元预算、想吃辣，有哪些组合
- 当前购物车怎么组合更省配送费
- 最近订单里哪些菜适合复购
- 哪些商家更快送达
- 为什么这次结算被拆成多个子单

这意味着后续 RAG 不会只是简单地把菜品文本扔进向量库，而是可以组合：

- 结构化过滤（地址、价格、商家状态、配送可达性）
- 向量召回（菜品描述、标签、用户偏好表达）
- 重排与推理（预算约束、人数约束、历史偏好、订单解释）

因此，第一期的价值在于把未来 agent 的“数据底盘”打牢，而不是只完成一个静态外卖页面。

---

## 11. 风险与约束

### 11.1 范围风险

如果第一期同时引入商家后台、真实支付、骑手系统或完整智能助手，会显著拉长周期并削弱主链路完成度。因此本设计坚持先做用户侧交易闭环。

### 11.2 数据一致性风险

跨商家结算会带来订单拆分和金额归集问题，因此第一期采用父单/子单 + 全有或全无策略，减少中间态复杂度。

### 11.3 后续兼容风险

如果第一期没有把地址、商家、菜品、购物车、订单的结构化关系建清楚，第二期助手和 RAG 会退化成浅层问答。因此第一期必须以可演化为目标。

---

## 12. 结论

第一期的最佳路径不是直接把智能助手做出来，而是先把“类美团外卖”的核心业务底座建成：

- 用户端交易链路完整
- 多商家结算模型正确
- 地址与配送逻辑可演示
- 商家与菜品数据足够真实
- 智能助手未来所需上下文接口已经预留

这样第二期再接入高质量 RAG 和智能助手时，系统才能真正体现出“基于真实业务上下文工作的 agent 应用”价值，而不是一个只有聊天框的 demo。
